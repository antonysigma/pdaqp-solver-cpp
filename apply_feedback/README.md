# apply_feedback

This submodule implements the affine transform of the selected active set of the
PDA-QP solver.

It consumes the `pdaqp.c`/`pdaqp.h` files generated by the CVXPyGen compiler, and repacks the generated matrices and vectors into **compile-time aligned C++ data structures**.

The primary goal is to allow modern compilers to:

- see the entire matrix-vector multiplication and addition at compile time;
- aggressively inline and fuse matrix–vector operations, via compiler argument `-O3`;
- emit SIMD-accelerated code on modern CPUs, via compiler argument `-march=native`;
- eliminate redundant multiplications, additions, and memory loads with zero-valued operands;

This approach allows the PDA-QP solver to retain the numerical behavior of
CVXPyGen while producing architecture-optimized machine code.

## Core idea

The PDA-QP algorithm separates the QP problem into two stages:

1. **Active set identification** A binary decision tree determines which set of
constraints is active for the current parameter vector.

2. **Direct affine evaluation** Once the active set is known, the optimal
solution can be computed using a pre-computed affine mapping.

Mathematically, each active set corresponds to a region of the parameter
space where the optimal solution takes the form

```
x = A @ p + b
```

The CVXPyGen tool (currently) emits these affine maps as generic C arrays.

This module reorganizes those matrices and vectors into compile-time C++
structures, allowing modern compilers to:

- propagate constant matrix coefficients
- fully unroll the matrix–vector operations
- eliminate zero multiplications
- emit SIMD-accelerated fused multiply-add instructions

The result is that evaluating the affine solution becomes a small, specialized
arithmetic kernel instead of a generic matrix multiply. For QP solvers with
explicit active set approach, the final step involves an affine transform:

## Compile-time architecture options

Two architectural strategies are supported.

### `applyFeedback<force_function_table = true>`

In this configuration, each affine operation is compiled into a separate specialized function. For small decimal types, e.g. 16-bit Q2.14 format, matrix and vector coefficients can be embedded directly as **immediate operands (IMM)** in the instruction stream.

Example:

```asm
0000000000000000 <auto (anonymous namespace)::pdaqp_solver_internal::applyFeedbackFn<1ul>(vector_math::Vector<(unsigned short)2, math::fixed<short, 14ul>> const&)>:
;     return fixed<int32_t, Q * 2>{static_cast<int32_t>(a.data) * b.data};
       0: 0f bf 07                      movswl  (%rdi), %eax
       3: 0f bf 57 02                   movswl  0x2(%rdi), %edx
       7: 69 c8 cc 0c 00 00             imull   $0xccc, %eax, %ecx      # imm = 0xCCC
       d: 69 f2 4a 16 00 00             imull   $0x164a, %edx, %esi     # imm = 0x164A
      13: 69 c2 5b fd ff ff             imull   $0xfffffd5b, %edx, %eax # imm = 0xFFFFFD5B
      19: 69 d2 28 17 00 00             imull   $0x1728, %edx, %edx     # imm = 0x1728
;     accu.data += b.data >> (Q - Q2);
      1f: c1 f9 02                      sarl    $0x2, %ecx
      22: c1 fe 02                      sarl    $0x2, %esi
      25: c1 f8 02                      sarl    $0x2, %eax
      28: c1 fa 02                      sarl    $0x2, %edx
      2b: 01 ce                         addl    %ecx, %esi
      2d: 01 c8                         addl    %ecx, %eax
;     return result;
      2f: 48 c1 e6 20                   shlq    $0x20, %rsi
      33: 48 09 f0                      orq     %rsi, %rax
; }
      36: c3                            retq

```

Key characteristics:

- coefficients embedded as immediates
- zero multiplications optimized out
- minimal memory access
- ideal for Harvard architectures and embedded MCUs

The compiler effectively turns the computation into pure, hardcoded arithmetic
instructions.

### `applyFeedback<force_function_table = false>`

In this configuration the solver uses **runtime matrix/vector lookup**.

The affine matrices and vectors are stored in the `rodata` section, and the
index `FeedbackID` selects the corresponding item using an O(1) table lookup.

```asm
cmpw    $0x2, %di
cmovbl  %edi, %eax

leaq    scale_list(%rip), %rcx
leaq    offset_list(%rip), %rdx

movswl  (%rdx,%rdi,4), %r10d
movswl  0x2(%rdx,%rdi,4), %eax
imull   %r11d, %r10d
[...]
```

Key characteristics:

- compact code size
- matrix data stored in `rodata`
- predictable O(1) access pattern
- suitable for large problems with many feedback regions

The compiler still generates SIMD acclerated instruction for the matrix-vector
computation.

## Why this approach works well today

Modern compilers are extremely capable at optimizing code when the data is
visible at compile time. By reorganizing the CVXPyGen output into compile-time
structures, the compiler can:

- constant-propagate matrix coefficients
- eliminate dead arithmetic
- vectorize fused multiply-add chains
- remove unnecessary memory loads

This effectively turns a generic affine control law into a specialized numeric
kernel generated by the C++ compiler itself.
